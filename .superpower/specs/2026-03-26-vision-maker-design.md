# vision-maker 设计方案

> 为新项目或已有项目建立项目自有文档体系的 AgentSkill。该文档体系帮助人/智能体对项目从顶层视角（概念、目标、要解决的问题、成功标准）到具体实现细节（代码、数据定义、产出内容等）建立完整认知。

---

## 一、Skill 定位与核心模型

### 1.1 Skill 的本质

vision-maker 是一套**思维模型 + 方法论**，不包含任何行业/领域的具体知识。它告诉智能体"如何为项目构建和维护文档体系"，而非"该写什么内容"。

```
vision-maker Skill = 思维模型 + 方法论（不变，Skill 内置）
                   + 项目元知识（变化，INIT 阶段产出，项目特定）
                   → 驱动文档体系的建立、维护、评审
```

### 1.2 核心循环

Skill 有四种运行模式，构成持续运转的认知循环：

```
INIT ──→ BRIEF ──→ MAINTAIN ──→ REVIEW
  ^                                 │
  └─────────────────────────────────┘
        （发现缺口时回到 INIT 补建）
```

| 模式 | 核心职责 |
|------|---------|
| **INIT** | 采集项目元知识，建立文档体系 |
| **BRIEF** | 按任务需求向开发者暴露完整的项目知识链 |
| **MAINTAIN** | 项目内容变更后，审慎检查并同步受影响的文档 |
| **REVIEW** | 按多维度评审文档体系或单个文档的质量 |

### 1.3 用户特征与适配策略

Skill 的用户（人类侧）具有以下典型特征，Skill 必须据此调整交互方式：

| 特征 | Skill 的适配策略 |
|------|-----------------|
| **知识面有限，存在认知盲区** | 默认将用户视为新手。通过用户提出的问题和输入的知识作为确定性证据，动态刻画其知识面范围，据此决定哪些内容简略描述、哪些详尽展开。 |
| **思维是散点，缺乏体系化** | 主动将用户零散的想法组织为能够支撑项目的结构化认知面，而非仅回应单个点。 |
| **方法论了解有限** | 在解决问题时主动上升到方法论层面进行解释和引导，除非用户明确要求跳过。 |
| **记忆有限，上下文易丢失** | 输出时始终补充完整上下文，不假设用户记得之前的讨论细节。跨会话时帮助用户快速恢复上下文。 |
| **决策疲劳** | 给出精简选项 + 明确推荐 + 推荐理由，而非罗列所有可能性。 |
| **锚定偏差** | 主动识别用户可能锚定的初始想法，暴露盲区和替代视角。 |
| **领域专家 ≠ 文档专家** | 用户可能非常了解自己的业务，但不擅长将隐性知识外化为结构化文档。Skill 应引导这个外化过程。 |

#### 用户知识画像

Skill 在交互过程中持续刻画用户的知识面，作为内容详略的依据：

- **确定性证据**：用户主动提出的问题（暴露未知）、用户输入的知识（确认已知）
- **刻画结果**存储在 `.vision/.meta/user.local.md` 中，不纳入版本控制
- 知识画像随交互持续更新，不是一次性评估

### 1.4 人机协作模型

智能体主导执行，人在关键节点审查和补充：

- INIT：智能体引导对话采集信息 → 人确认元知识和文档蓝图
- BRIEF：智能体自主组装上下文 → 人/智能体消费
- MAINTAIN：智能体检测变更并提出修改建议 → 人审批
- REVIEW：智能体执行评审 → 人确认评审结论和改进措施

## 二、`.vision/` 目录结构

```
.vision/
├── .meta/                           # 元信息层（Skill 的工作上下文）
│   ├── knowledge.md            # 项目元知识（纳入版本控制）
│   └── user.local.md               # 用户个人特征画像（不纳入版本控制）
├── VISION.md                        # 项目顶层文档（文档图的根节点）
├── <layer-name>/                    # 语义化的层级目录（由元知识定义）
│   ├── <topic>.md
│   ├── adr/                         # 架构决策记录（如果该层涉及架构）
│   │   └── 001-<decision>.md
│   └── <topic>/
│       ├── <document>.md
│       └── references/              # 附属资源（Tier 3，按需加载）
│           ├── diagrams/
│           └── schemas/
└── <layer-name>/
    └── ...
```

版本控制规则：

```gitignore
# .vision/.meta/ 下的 *.local.md 不纳入版本控制
.vision/.meta/*.local.md
```

核心原则：

- `.vision/` 下有三类东西：**元信息层**（`.meta/`）+ **顶层文档**（`VISION.md`）+ **文档体系本身**
- `.meta/` 目录存放 Skill 工作所需的元信息，其中项目元知识纳入版本控制，用户个人画像不纳入
- 层级目录的名称由 `knowledge.md` 中的文档粒度定义决定，必须具有语义（不使用 L0/L1/L2 等无语义编号）
- 文档载体为 Markdown，元数据通过 front-matter（YAML）承载

## 三、元信息层 `.vision/.meta/`

### 3.1 定位

`.meta/` 目录是 Skill 工作所需的元信息存储层，包含两份文件：

| 文件 | 用途 | 版本控制 |
|------|------|:--------:|
| `knowledge.md` | 项目元知识：行业、领域、场景、概念体系、文档粒度定义、评审维度 | 是 |
| `user.local.md` | 用户个人特征画像：知识面范围、沟通偏好、认知盲区记录 | 否 |

### 3.2 项目元知识 `knowledge.md`

元知识是 Skill 在当前项目中工作的**上下文锚点**。Skill 的方法论是通用的，元知识使其适配到具体项目。

### 3.3 knowledge.md 内容结构

```yaml
---
project_name: "项目名称"
industry: "所属行业"
domain: "问题领域"
---
```

> front-matter 字段供机器快速读取（如 REVIEW 扫描），正文供人/智能体深度理解。两者存在语义重叠时，正文为源头，front-matter 为摘要。

正文包含以下章节（由 INIT 阶段通过对话产出）：

**行业与领域**
- 项目所属行业的特征描述
- 项目解决的核心问题领域
- 行业内的关键约束和规范要求

**核心场景**
- 项目面对的核心使用场景列表
- 每个场景的简要描述

**领域概念体系**
- 行业/领域内的关键术语和定义
- 概念之间的关系

**文档粒度定义**
- 文档体系分几层，每层的语义名称和职责
- 每层文档面向的消费者和认知目标
- 示例：

  | 层级目录 | 职责 | 面向 |
  |---------|------|------|
  | `architecture/` | 系统架构、技术选型、关键设计决策 | 需要理解"为什么这样设计"的开发者 |
  | `modules/` | 每个模块的职责、接口、数据结构、关键流程 | 需要在具体模块中工作的开发者 |

**评审维度**
- 基础维度（Skill 内置，适用所有项目）+ 项目特定维度
- 每个维度的检查标准
- 参见第八节 REVIEW 模式

**项目特定约束**
- 团队规范、技术栈约束、合规要求等

### 3.4 元知识的更新

当项目方向调整或领域认知深化时，通过 REINIT 更新 `knowledge.md`，并触发 MAINTAIN 评估级联影响。

### 3.5 用户画像 `user.local.md`

存储 Skill 对当前用户的认知，用于适配交互方式和内容详略。

```yaml
---
last_updated: 2026-03-26
---
```

正文由 Skill 在交互过程中持续维护，包含：

**知识面画像**
- 已确认的知识领域（用户主动输入的知识作为证据）
- 已识别的知识盲区（用户提出的问题作为证据）
- 需要详尽解释的领域 / 可以简略带过的领域

**沟通偏好**
- 用户偏好的沟通风格和详略程度
- 用户明确提出的工作习惯

**交互记录摘要**
- 关键讨论结论的简要索引（帮助跨会话恢复上下文）

> `user.local.md` 是个人文件，不纳入版本控制。多人协作时每人维护自己的画像。

## 四、项目顶层文档 `VISION.md`

### 4.1 定位

`VISION.md` 是文档体系的**根节点**，也是所有消费者（人/智能体）认知项目的**入口**。

### 4.2 内容结构

```yaml
---
description: "一句话描述项目的核心价值"
concepts: [项目核心概念列表]
children:
  - architecture/overview.md
  - modules/overview.md
---
```

正文包含：

- **项目是什么**：一段话概述
- **为什么存在**：要解决的问题
- **目标与成功标准**：可衡量的目标
- **核心概念**：项目中最重要的几个概念及其关系
- **文档体系导览**：简要说明 `.vision/` 的层级结构和导航方式

## 五、文档 Front-matter 规范

每份文档（除 `knowledge.md` 外）的 front-matter 遵循统一规范：

```yaml
---
description: "用一两句话摘要本文档的内容和用途（< 100 tokens）"
type: explanation | reference | guide | context
concepts: [本文档涉及的核心概念列表]
depends_on:
  - path/to/prerequisite-doc.md       # 理解本文档的前置知识（向上）
children:
  - path/to/more-detailed-doc.md      # 本文档展开的更细粒度文档（向下）
referenced_by:
  - path/to/related-doc.md            # 非上下级，但存在概念关联的文档（横向）
last_verified: 2026-03-26
---
```

### 5.1 字段说明

| 字段 | 必须 | 用途 |
|------|:----:|------|
| `description` | 是 | Tier 1 发现层：轻量摘要，供快速 grep 定位 |
| `type` | 是 | 文档分类，借鉴 Diataxis |
| `concepts` | 是 | 核心概念标签，支持 concepts 匹配和 MAINTAIN 影响分析 |
| `depends_on` | 否 | 知识的上下级关系（向上）：理解本文档需要的前置知识 |
| `children` | 否 | 知识的上下级关系（向下）：本文档展开的更细节文档 |
| `referenced_by` | 否 | 引用关系（横向）：非上下级但有概念关联 |
| `last_verified` | 是 | 最后验证日期，供 REVIEW 时效性检查 |

### 5.2 文档类型（type）

借鉴 Diataxis 框架，适配文档体系场景：

| 类型 | 语义 | 示例 |
|------|------|------|
| `explanation` | 解释概念和原因（为什么） | 架构决策记录、设计理由 |
| `reference` | 精确的事实查阅（是什么） | API 定义、数据模型、配置项 |
| `guide` | 完成特定任务的步骤（怎么做） | 本地环境搭建、部署流程 |
| `context` | 智能体专用的任务上下文片段 | BRIEF 模式组装的上下文包 |

### 5.3 三种关系类型

文档间通过 front-matter 形成一个**有向图**：

| 关系 | 方向 | 语义 | 用途 |
|------|------|------|------|
| `depends_on` | 向上 | 知识的上下级：前置知识 | BRIEF 上溯加载完整知识链 |
| `children` | 向下 | 知识的上下级：细粒度展开 | 自顶向下遍历、影响面分析 |
| `referenced_by` | 横向 | 引用关系：概念关联 | 补充关联知识、影响面扩散 |

所有路径均为相对于 `.vision/` 目录的相对路径。

## 六、三级加载协议

借鉴 AgentSkill 的渐进式暴露思想，适配文档体系的使用场景：

```
Tier 1 — 发现（极低成本）
  扫描文档的 description + concepts 字段
  判断文档是否与当前任务相关
  开销：每文档 ~1-2 行文本

Tier 2 — 加载（受控成本）
  读取文档正文
  每份文档建议 < 5000 tokens
  沿关系字段按需遍历其他文档

Tier 3 — 深入（按需）
  文档正文内以内联引用方式指向的附属资源
  如详细序列图、完整 API schema、代码示例等
  仅在智能体判断需要时才加载
```

### 6.1 附属资源机制（Tier 3）

文档正文保持精炼，重型内容外置为附属资源：

```
modules/auth/
├── session-lifecycle.md          # Tier 2：正文，< 5000 tokens
└── references/
    ├── diagrams/
    │   └── token-refresh-flow.svg   # Tier 3：按需加载
    └── schemas/
        └── session-model.json       # Tier 3：按需加载
```

正文中通过内联引用指向附属资源：

```markdown
会话刷新的完整流程参见 [Token 刷新序列图](./references/diagrams/token-refresh-flow.svg)。
```

### 6.2 加载策略与关系遍历

**BRIEF 模式下的加载不设人为深度限制**。当智能体需要开发某个功能时，应沿 `depends_on` 完整上溯，获取从概念到细节的完整知识链。加载范围由任务需求决定，而非由固定深度约束。

三级协议控制的是**每份文档内部**的加载粒度（description → 正文 → 附属资源），而非文档间的遍历深度。

## 七、四种运行模式详细流程

### 7.1 INIT 模式 — 建立认知

**触发时机**：新项目首次使用 / 已有项目首次接入 / 新模块加入 / 元知识更新（REINIT）

> 各模式的触发方式由项目集成环境决定（人手动触发、CI 集成等），Skill 本身定义"做什么"，不绑定特定的触发机制。

#### 场景 1.1：元知识采集 + 顶层认知建立

```
步骤：
1. 通过与人的对话，了解项目的行业、领域、核心场景、关键概念
   （同时根据对话内容初始化 user.local.md，刻画用户知识面）
2. 确定文档体系的分层方案（几层、每层语义名称和职责）
3. 确定项目特定的评审维度
4. 产出 .meta/knowledge.md → 人审批
5. 产出 VISION.md → 人审批
```

#### 场景 1.2：新项目文档冷启动

```
步骤：
1. 基于元知识中的分层定义，创建目录结构
2. 确定每层需要哪些文档（文档蓝图）→ 人审批
3. 按优先级逐份生成文档，每份生成后人审批
4. 建立文档间的 depends_on / children / referenced_by 关系
5. 运行 REVIEW 检查初始文档体系质量
```

#### 场景 1.3：已有项目补建

```
步骤：
1. 完成场景 1.1（元知识采集）
2. 分析现有代码结构、README、注释、配置文件
3. 将分析结果与元知识对照，识别需要哪些文档
4. 按场景 1.2 的步骤 2-5 生成文档
```

#### 场景 1.4：新模块接入

```
步骤：
1. 读取现有文档体系结构
2. 确定新模块文档在体系中的位置
3. 生成新模块文档 → 人审批
4. 更新相关文档的关系字段（depends_on / children / referenced_by）
```

#### 场景 1.5：元知识更新（REINIT）

```
步骤：
1. 更新 knowledge.md → 人审批
2. 评估变更对现有文档体系的影响（触发 MAINTAIN 3.4）
```

### 7.2 BRIEF 模式 — 支撑开发

**核心职责**：按任务需求，向开发者（人/智能体）暴露完整的项目知识链。

#### 导航策略

**策略 A：自顶向下遍历（Onboarding 场景）**

```
VISION.md → 沿 children 逐层展开 → 按需停止
适用于：首次接触项目，需要建立全局心智模型
```

**策略 B：概念定位 + 关系遍历（日常开发场景）**

```
1. Tier 1 发现：grep concepts/description 定位目标文档
2. Tier 2 加载：读取目标文档正文
3. 沿 depends_on 完整上溯：获取前置知识链
4. 沿 referenced_by 补充：获取横向关联知识
5. Tier 3 按需：加载正文中引用的附属资源
```

**策略 C：反向追溯（Code Review / 影响面分析场景）**

```
1. 从变更文件/目标模块出发，定位相关文档
2. 沿 depends_on 上溯获取设计理由
3. 沿 children + referenced_by 正向扩散评估影响面
```

#### 场景明细

| 场景 | 导航策略 | 说明 |
|------|---------|------|
| 项目 Onboarding | 策略 A | 自顶向下，广度优先 |
| 开发具体功能 | 策略 B | 定位 → 完整上溯 → 补充关联 |
| 修复 Bug | 策略 B | 定位模块文档 → 上溯依赖链 |
| Code Review | 策略 C | 从变更反查设计意图 |
| 架构决策评估 | 策略 B | 定位架构文档 + 相关 ADR |
| 影响面分析 | 策略 C | 从目标文档正向扩散 |

### 7.3 MAINTAIN 模式 — 同步认知

**核心职责**：当项目内容发生变更后，审慎检查变更对文档体系的影响，调整受影响的文档内容，使文档体系与项目实际状态始终保持一致。

#### 工作流程

```
1. 识别变更范围
   - 分析代码变更涉及了哪些 concepts
   - 通过 concepts 匹配定位可能受影响的文档

2. 评估影响
   - 逐份检查受影响文档的内容是否仍然准确
   - 判断影响级别：内容需更新 / 关系需调整 / 文档需拆分或合并 / 需新增文档

3. 执行修正
   - 更新文档内容
   - 调整 depends_on / children / referenced_by 关系
   - 校验双向关系一致性（如 A 新增 children B，则 B 的 depends_on 须包含 A）
   - 更新 last_verified 日期

4. 人审批
   - 将修改建议呈现给人确认
```

#### 场景明细

| 场景 | 触发条件 | 动作 |
|------|---------|------|
| 代码变更后文档同步 | 代码提交 / 功能合并 | concepts 匹配 → 检查文档准确性 → 修正偏离 |
| 架构变更后文档重组 | 模块拆分/合并/重命名 | 文档拆分/合并 → 重建关系 → 更新内容 |
| 新增决策记录 | 做出重要技术决策 | 创建 ADR → 更新相关文档引用 |
| 元知识变更的级联影响 | REINIT 更新了元知识 | 评估对现有文档的影响 → 调整内容和评审标准 |

### 7.4 REVIEW 模式 — 评审质量

**核心职责**：检查文档体系整体或单个文档的生成质量。

#### 评审维度

**基础维度（Skill 内置，适用所有项目）：**

| 维度 | 检查内容 |
|------|---------|
| **必要性** | 这份文档是否有存在的理由？是否有消费者需要它？ |
| **完整性** | 文档是否覆盖了它声称要覆盖的 concepts？是否有遗漏？ |
| **准确性** | 文档内容是否与当前代码/架构一致？ |
| **可参考性** | 文档是否结构清晰、易于被人/智能体理解和使用？ |
| **关系完整性** | depends_on / children / referenced_by 是否有效？是否有孤立文档？ |

**项目特定维度（从元知识派生）：**

由 `knowledge.md` 中的"评审维度"章节定义。例如：

- 金融项目：合规性维度
- 医疗项目：数据隐私维度
- 开源项目：贡献者友好度维度

#### 评审范围

| 范围 | 说明 |
|------|------|
| 单文档评审 | 按所有维度逐项检查一份文档，输出评审报告 |
| 模块级评审 | 检查一个模块所有文档的整体覆盖度和一致性 |
| 体系级评审 | 覆盖度、时效性、关系完整性、孤立文档检测 |
| 元知识评审 | 检查 knowledge.md 是否仍反映项目现状 |

#### 关系完整性检查

REVIEW 模式负责检测文档图的健康状态：

- `depends_on` / `children` / `referenced_by` 指向的文件是否存在
- 是否有孤立文档（无任何关系连接）
- 是否有循环依赖
- 双向关系是否一致（A 的 children 包含 B，B 的 depends_on 是否包含 A）

## 九、设计原则总结

| 原则 | 说明 |
|------|------|
| **Skill 是方法论，不是模板** | Skill 提供思维框架，具体内容由元知识驱动 |
| **元信息层与文档体系分离** | `.meta/` 指导 Skill 工作，文档体系是面向消费者的产出 |
| **适配用户认知水平** | 动态刻画用户知识面，据此调整内容详略；主动暴露盲区和方法论；输出始终包含完整上下文 |
| **散点组织成面** | 将用户零散的想法主动组织为结构化的项目认知，而非仅回应单个点 |
| **文档自描述、自导航** | 每份文档通过 front-matter 携带元数据和关系，不依赖外部索引 |
| **三级加载控制上下文** | description → 正文 → 附属资源，控制单文档的加载粒度 |
| **关系遍历不设深度限制** | BRIEF 场景按任务需求完整加载知识链 |
| **语义化组织** | 目录名称由元知识定义，反映项目实际的认知粒度 |
| **人机协作，关键节点门控** | 智能体主导执行，人在产出物审批节点把关 |
| **文档与项目状态保持一致** | MAINTAIN 确保文档体系是活的，不是一次性产物 |
| **评审维度随项目适配** | 基础维度 + 元知识派生的项目特定维度 |
| **个人数据不入版本控制** | 用户画像（`user.local.md`）属于个人，通过 `.gitignore` 排除 |
