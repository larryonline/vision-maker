# Vision Maker 响应协议设计

> 日期：2026-03-27
> 状态：已确认，待实施

## 1. 背景与根因

### 1.1 问题现象

用户首次使用 vision-maker 时面临的核心困惑：**"我不知道该输入什么内容才能让 vision-maker 开始工作"**。

通过系统性的深度反思和盲点发掘，我们发现问题的根因在于：

> **SKILL.md 定义了模式的职责（做什么），但没有定义响应协议（如何被触发、如何衔接、如何自检）**。

### 1.2 设计冲突

原始设计中存在多处矛盾：
- MAINTAIN 和 REVIEW 都评估"准确性"，但范围界定方式不同
- MAINTAIN 只接受"项目变更"触发，无法处理 REVIEW 发现的既有问题
- INIT 既做"思考"（采集知识、设计蓝图）又做"执行"（创建文件），职责过宽
- BRIEF 作为独立模式的定位模糊——它本质上是文档消费协议

## 2. 改进目标

1. 为每个模式补充完整的响应协议（触发/输入/深度/输出/衔接）
2. 修正模式设计中的职责重叠和工作流断裂
3. 分离"思考"与"执行"——模式是认知框架，执行由方法论引导的任务完成
4. 支持自动化集成（git hooks、CI），同时保持人工触发能力

## 3. 架构总览

```
vision-maker SKILL
│
├── 三个认知框架（模式）
│   ├── INIT    建立文档体系 + 配置自动化集成
│   ├── BRIEF   获取项目知识（已有 + 即时补充）
│   └── AUDIT   评估文档质量和一致性（合并原 MAINTAIN + REVIEW）
│
├── 三个方法论（共享引用）
│   ├── knowledge-acquisition.md   知识获取与管理方法论
│   ├── doc-methodology.md         文档工程方法论
│   └── doc-tasks.md               文档操作任务规格
│
├── 跨模式机制
│   ├── 响应协议规范（每个模式的交互契约）
│   ├── 项目侧配置自检（所有模式的前置流程）
│   ├── 一致性标记（品牌化 commit message 审计标记）
│   └── 衔接协议（模式完成后的推荐下一步 + .vision 提示词）
│
├── scripts/
│   └── validate.sh   结构性快检
│
└── assets/templates/
    └── 扩展的模板文件
```

## 4. 三级加载协议

| 层级 | 内容 | 位置 |
|------|------|------|
| Tier 1 | 模式概览 + 触发关键词 | SKILL.md description |
| Tier 2 | 模式路由 + 响应协议摘要 + 自检流程 + doc-tasks 路由入口 | SKILL.md body（< 500 行） |
| Tier 3 | 模式详细流程、方法论、任务规格 | references/ |

## 5. 模式设计

### 5.1 INIT — 建立文档体系

**设计定位**：引导人和智能体思考"项目需要什么样的文档体系"，并配置自动化守护。

**响应协议**

| 项 | 定义 |
|---|---|
| 触发方式 | 人工 / 自检发现无 .vision/ 时引导 |
| 触发示例 | "为当前项目建立 .vision 文档体系" / "用 vision-maker 初始化这个项目" |
| 输入 | 项目代码结构 + 对话采集的领域知识 |
| 执行深度 | 新项目全流程 / 已有项目补建 / 新模块接入 / 元知识更新 |
| 输出 | .vision/ 文档体系 + .meta/integration.md + hooks/CI 配置（需用户许可） |
| 衔接 | 推荐 AUDIT 验证 + 告知 BRIEF 使用方式 + 询问是否写入 CLAUDE.md/AGENTS.md |

**关键改进**：
- INIT 对话采集新增"领域文档标准"：基于行业和项目类型，智能体主动推导应覆盖的文档关注面
- 文档创建步骤引用 doc-tasks.md §创建文档
- 自动化集成配置作为 INIT 的产出之一

### 5.2 BRIEF — 获取项目知识

**设计定位**：为开发任务组装前置知识链，嵌入智能体的研究阶段。

**响应协议**

| 项 | 定义 |
|---|---|
| 触发方式 | 人工显式触发 / 智能体研究阶段调起 |
| 触发示例 | "使用 .vision 获取 XX 的所有知识" / "去 .vision 里找 XX 相关资料" |
| 输入 | 任务描述 + 关联文件（触发内容提取 / 智能体自动识别） |
| 执行深度 | 策略 A 自顶向下 / 策略 B 概念定位 / 策略 C 反向追溯 |
| 输出 | 知识链（标注来源和置信度）+ 知识盲区标注 |
| 衔接 | 交还控制权 + 盲区建议行动提示词 |

**关键改进**：
- BRIEF 不再是纯粹的"读取已有文档"，遇到知识盲区时按 knowledge-acquisition.md 的优先级链即时补充
- 即时补充的知识标记置信度（✓ 已验证 / ~ 推断 / ? 未知），建议正式写入 .vision/

### 5.3 AUDIT — 评估文档质量和一致性

**设计定位**：合并原 MAINTAIN + REVIEW。对文档体系进行一致性和质量评估，覆盖所有评估维度和触发场景。

**合并理由**：准确性评估逻辑重复；"一致性"是"质量"的子集；区别仅在范围界定方式，应作为参数而非两个模式。

**响应协议**

| 项 | 定义 |
|---|---|
| 触发方式 | 自动（pre-commit hook）/ 人工 |
| 触发示例 | "评审 .vision 文档质量" / "代码改完了，检查文档一致性" / "对 .vision 做全面体检" |
| 输入 | 变更驱动：git diff + 追溯未检查提交；人工：用户指定范围 |
| 评估维度 | 准确性 + 完整性 + 必要性 + 可参考性 + 关系完整性 + 项目特定维度 |
| 执行深度 | 见下表 |
| 输出 | 诊断报告（问题分类 + 修复建议）/ pass 标记 |
| 衔接 | 有问题：推荐修复提示词；无问题：推荐下次审计时机 |

**执行深度**

| 触发场景 | 范围界定 | 维度 | 深度 | 产出 |
|---------|---------|------|------|------|
| pre-commit 自动 | 变更文件→受影响文档+追溯 | 准确性+关系+变更完整性 | 仅诊断 | pass/fail + 报告 |
| 人工一致性检查 | 用户指定或变更驱动 | 准确性+关系+完整性 | 诊断+修复方案 | 报告→doc-tasks修复 |
| 人工全面评审 | 单文档/模块/体系/元知识 | 全维度 | 深度评审 | 评审报告 |
| INIT 后验证 | 全体系 | 全维度 | 诊断 | 评审报告 |

**评估范围界定规则**：
1. 变更文件 → 提取 concepts → 匹配 .vision/ 文档
2. 沿 depends_on / referenced_by 扩展一层
3. 追溯：本次变更 + 向前至上一个审计通过标记
4. 全量评估触发条件（满足任一）: 首次执行 / 追溯超过5次仍无标记 / 累计变更涉及20%以上文件

## 6. 方法论引用文件

### 6.1 knowledge-acquisition.md

知识获取与管理方法论：
- 研究策略：如何从源码中定位和提取知识
- 追溯策略：如何从 git history / PR / commit message 中恢复设计决策
- 置信度框架：✓ 已验证 / ~ 推断 / ? 未知
- 对话获取策略：何时问用户、如何提问
- 知识回流规则：任务中获得的新知识如何流回 .vision/

### 6.2 doc-methodology.md

文档工程方法论：
- 文档结构标准
- 文档大小控制
- 关系建立规则
- concepts 标签规范
- 层级结构原则

### 6.3 doc-tasks.md

文档操作任务规格。文件开头包含执行方式提示：

> 以下每个任务规格是一个自治执行单元。如果当前平台支持 sub-agent 能力，建议将任务规格作为 sub-agent 的 system prompt 派发执行。

**§创建文档**
```
角色：深入理解项目上下文的文档工程师
方法论：knowledge-acquisition.md + doc-methodology.md
输入：目标概念/模块 + 相关源码位置 + knowledge.md + 现有 .vision/ 文档图
步骤：研究源码 → 提取知识 → 确定位置 → 撰写文档 → 建立关系 → 校验 → 人审批
```

**§修复文档**
```
角色：确保文档与项目一致的质量工程师
输入：AUDIT 问题报告 + 相关源码 + 当前文档内容
步骤：理解问题 → 确认源码状态 → 执行修正 → 更新 last_verified → 校验 → 人审批
```

## 7. 跨模式机制

### 7.1 项目侧配置自检

任何模式启动时执行，通过智能体能力 + scripts/validate.sh 完成：
1. .vision/ 目录存在？
2. .meta/knowledge.md 存在且非空？
3. .meta/integration.md 存在？
4. 自动化集成就绪？
5. user.local.md 中智能体配置完整？

项目可在 integration.md 中注册自定义检测规则。

### 7.2 一致性标记

品牌化 commit message footer：

| 场景 | 标记 |
|------|------|
| 发现偏差并已对齐 | 🎯 vision-maker 检测到本次提交影响了 {N} 份文档，已帮您完成对齐。 |
| 审计通过，无需变更 | ✅ vision-maker 已仔细审计本次提交，文档与项目内容完全一致。 |
| 无受影响文档 | 👁 vision-maker 扫描了本次变更，未发现需要更新的文档。 |
| 全量审计通过 | 🔍 vision-maker 对本次提交进行了全量审计，所有文档均已与项目保持同步。 |

### 7.3 衔接协议

每个模式完成后：
1. 推荐下一步行动
2. 提供与本次任务相关的 .vision 提示词示例
3. 如检测到 CLAUDE.md / AGENTS.md 存在，询问用户是否写入 .vision 使用建议（需明确许可）

## 8. SKILL 与 .vision 的职责分离

| SKILL 侧 | .vision 侧 |
|---|---|
| 模式定义 + 响应协议 | 文档体系 |
| knowledge-acquisition.md | .meta/knowledge.md（含领域文档标准） |
| doc-methodology.md | .meta/integration.md（自动化集成配置） |
| doc-tasks.md | .meta/user.local.md（用户画像+智能体配置） |
| 自检流程规范 | .meta/navigation-guide.md（无 vision-maker 导航指南） |
| 一致性标记规范 | 实际的 hooks / CI workflow 文件 |

## 9. SKILL.md 路由表

```
用户意图                              路由目标
──────────────────────────────────    ──────────────
建立文档体系 / 初始化 / 接入          → INIT (init-mode.md)
获取知识 / .vision 查找资料            → BRIEF (brief-mode.md)
评审质量 / 检查一致性 / 审计           → AUDIT (audit-mode.md)
创建/修改/删除/重组文档                → doc-tasks.md（对应任务规格）
意图不明确                             → 自检 → 状态诊断 → 推荐模式
```
