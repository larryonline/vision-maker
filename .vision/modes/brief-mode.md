# BRIEF 模式 — 知识导航

## 设计定位

按任务需求暴露完整知识链，即时补充盲区。是智能体在开发任务中获取项目上下文的主要入口。

## 响应协议

| 项 | 定义 |
|---|------|
| 触发方式 | 人工 / 智能体研究阶段 |
| 触发示例 | "获取 XX 知识"、"去 .vision 找 XX"、"理解项目" |
| 输入 | 任务描述 + 关联文件 |
| 输出 | 知识链（来源 + 置信度）+ 盲区标注 |
| 衔接 | 交还控制权 + 盲区行动建议 |

## 核心流程

```
1. 接收任务描述
2. 搜索 .vision/ 文档（通过 concepts、description）
3. 沿 depends_on 上溯获取知识链
4. 标记置信度（✓ / ~ / ?）
5. 识别盲区并标注
6. 输出知识链 + 盲区
```

## 搜索策略

1. **concepts 匹配**：在 front-matter concepts 中搜索相关标签
2. **description 匹配**：在 front-matter description 中搜索关键词
3. **全文搜索**：在文档正文中搜索相关内容

## 知识链组装

沿 depends_on 完整上溯，不设深度限制：

```
任务文档 → 依赖文档 → 依赖的依赖 → ... → VISION.md
```

每层加载时遵循三级加载协议：
- Tier 1：先扫描 description + concepts
- Tier 2：加载相关文档正文
- Tier 3：按需加载附属资源

## 置信度标记

| 标记 | 含义 | 适用场景 |
|------|------|---------|
| ✓ 已验证 | 通过代码/文档/对话确认 | 源码直接可验证、用户明确确认 |
| ~ 推断 | 基于上下文合理推断 | 命名约定推断、模式推断 |
| ? 未知 | 无法确认，需要进一步核实 | 文档过时、代码与文档矛盾 |

## 盲区处理

当发现文档无法覆盖任务需求时：
1. 标注为盲区
2. 建议创建新文档来覆盖
3. 如用户确认，引用 doc-tasks.md 执行创建

## 输出格式

```
## 知识链

### [文档标题] (✓/~/?)
- 来源：.vision/path/to/doc.md
- 核心要点：...
- 置信度理由：...

## 盲区

| 缺失知识 | 建议行动 |
|---------|---------|
| XX | 创建新文档 / 补充到现有文档 |
```
