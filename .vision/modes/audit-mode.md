# AUDIT 模式 — 评估文档质量和一致性

## 设计定位

合并原 MAINTAIN + REVIEW，对文档体系进行一致性和质量评估，覆盖所有评估维度和触发场景。

**合并理由**：准确性评估逻辑重复；"一致性"是"质量"的子集；区别仅在范围界定方式。

## 响应协议

| 项 | 定义 |
|---|------|
| 触发方式 | 自动（pre-commit hook）/ 人工 |
| 输入 | 变更驱动：git diff + 追溯 / 人工：用户指定范围 |
| 输出 | 诊断报告（问题分类 + 修复建议）/ pass 标记 |
| 衔接 | 有问题：推荐修复提示词；无问题：推荐下次审计时机 |

## 执行深度

| 触发场景 | 范围界定 | 维度 | 深度 | 产出 |
|---------|---------|------|------|------|
| pre-commit 自动 | 变更文件→受影响文档+追溯 | 准确性+关系+变更完整性 | 仅诊断 | pass/fail + 报告 |
| 人工一致性检查 | 用户指定或变更驱动 | 准确性+关系+完整性 | 诊断+修复方案 | 报告→doc-tasks修复 |
| 人工全面评审 | 单文档/模块/体系/元知识 | 全维度 | 深度评审 | 评审报告 |
| INIT 后验证 | 全体系 | 全维度 | 诊断 | 评审报告 |

## 评估维度

### 基础维度（所有项目通用）

| 维度 | 检查内容 |
|------|---------|
| **必要性** | 文档是否有存在的理由？是否有消费者需要它？ |
| **完整性** | 文档是否覆盖了 concepts 字段声称的所有概念？ |
| **准确性** | 内容是否与当前代码/架构一致？ |
| **可参考性** | 结构是否清晰？能否快速找到所需信息？ |
| **关系完整性** | depends_on/children/referenced_by 是否有效且双向一致？ |

### 项目特定维度

由 knowledge.md 中的"评审维度"章节定义。本项目（vision-maker）的特定维度：
- **AgentSkill 规范合规性**：SKILL.md 结构、front-matter 约束、文件组织
- **上下文效率**：文档大小 < 5000 tokens
- **渐进式暴露**：信息按 Tier 1/2/3 合理分层
