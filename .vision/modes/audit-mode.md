# AUDIT 模式 — 文档审计

## 设计定位

评估文档体系的准确性、完整性、一致性等多维度质量，确保文档与项目内容保持同步。

## 响应协议

| 项 | 定义 |
|---|------|
| 触发方式 | 自动（pre-commit）/ 人工 |
| 触发示例 | "评审文档质量"、"检查一致性"、"对 .vision 做体检" |
| 输入 | git diff + 追溯 / 用户指定范围 |
| 输出 | 诊断报告 / pass 标记 |
| 衔接 | 修复提示词 / 下次审计时机 |

## 审计维度

### 基础维度（内置）

| 维度 | 检查内容 |
|------|---------|
| 必要性 | 文档是否有消费者？是否冗余？ |
| 完整性 | 知识覆盖是否有盲区？ |
| 准确性 | 文档内容是否与代码/配置一致？ |
| 可参考性 | 文档是否易于理解和使用？ |
| 关系完整性 | depends_on/children/referenced_by 是否一致？有无断链？ |

### 项目特定维度

由 knowledge.md 中的"评审维度 → 项目特定维度"定义。

## 审计范围

| 范围 | 说明 |
|------|------|
| changed | 仅审计 git diff 中变更的文件及其关联文档 |
| full | 全量审计所有文档 |

## 核心流程

```
1. 确定审计范围（changed / full）
2. 对每个文档执行维度检查
3. 发现偏差时：
   a. 检查源码当前状态
   b. 与文档内容对比
   c. 确定正确的描述
   d. 评估修复策略
4. 生成诊断报告
5. 如为自动模式且启用自动修复，执行修复
```

## 诊断报告格式

```
## 审计报告

范围：changed / full
时间：YYYY-MM-DD HH:MM

### 通过项
- ✓ [文档] 必要性检查通过
- ✓ [文档] 完整性检查通过

### 问题项
- ✗ [文档] 准确性问题：描述与代码不一致
  - 期望：...
  - 实际：...
  - 修复建议：...

### 统计
- 审计文档数：N
- 通过：N
- 问题：N
```

## 一致性标记

根据审计结果，在 commit message 中添加对应标记：

| 场景 | 标记 |
|------|------|
| 发现偏差并已对齐 | 🎯 vision-maker 检测到本次提交影响了 {N} 份文档，已帮您完成对齐。 |
| 审计通过，无需变更 | ✅ vision-maker 已仔细审计本次提交，文档与项目内容完全一致。 |
| 无受影响文档 | 👁 vision-maker 扫描了本次变更，未发现需要更新的文档。 |
| 全量审计通过 | 🔍 vision-maker 对本次提交进行了全量审计，所有文档均已与项目保持同步。 |

## 元知识变更的级联影响

当 knowledge.md 发生变更时：
1. 重新评估文档粒度定义是否影响现有文档结构
2. 检查新增/变更的概念是否需要新文档覆盖
3. 验证评审维度变更是否影响现有文档评估
