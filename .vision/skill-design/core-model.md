# 核心模型

## 三要素架构

```
vision-maker = 思维模型（如何思考文档）
             + 方法论（如何操作文档）
             + 项目元知识（项目特定上下文）
             → 驱动文档体系的建立、维护、评审
```

**思维模型**和**方法论**由 Skill 本身承载，跨项目不变。**项目元知识**存储在 `.vision/.meta/knowledge.md`，是框架具体化的驱动力。

这意味着 vision-maker 不包含任何行业/领域的具体知识——它是一个通用的认知框架，告诉智能体"如何为项目构建和维护文档体系"，而非"该写什么内容"。

## 方法论不是模板

vision-maker 提供的是思维框架，不是文档模板。模板（`assets/templates/`）仅提供 front-matter 骨架，正文内容完全由元知识驱动、在对话中与用户共同产出。

## 三种运行模式

模式是方法论的操作化（认知框架），每个模式定义了完整的**响应协议**（触发/输入/输出/衔接）：

| 模式 | 认知语义 | 何时触发 |
|------|---------|---------|
| **INIT** | 建立文档体系 | 项目/模块首次接入，或元知识需要更新 |
| **BRIEF** | 获取项目知识 | 开发、调试、审查等需要理解项目的任务 |
| **AUDIT** | 评估文档质量 | 代码变更后、定期检查、INIT 后验证 |

三种模式构成认知循环：INIT 建立 → BRIEF 消费 → AUDIT 评审 → 发现问题可能触发 INIT。

## 思考与执行分离

**模式是认知框架**（教你怎么想）——定义触发方式、输入输出、衔接协议。
**任务规格是执行指南**（教你怎么做）——定义在 doc-tasks.md 中，是可派发的自治执行单元。

这种分离确保了：
- 模式保持简洁，聚焦认知语义
- 任务规格可独立演进和组合
- 支持 sub-agent 派发执行

## 元知识的角色

元知识（knowledge.md）决定了框架如何具体化：

- **文档粒度**：几层目录、每层的语义名称 → 决定 `.vision/` 的目录结构
- **领域概念**：概念体系和术语 → 决定文档的 concepts 标签体系
- **评审维度**：基础维度 + 项目特定维度 → 决定 AUDIT 的检查标准
- **核心场景**：项目面临的使用场景 → 影响文档覆盖范围的优先级
- **领域文档标准**：基于行业的文档关注面 → 指导 INIT 阶段的内容覆盖
