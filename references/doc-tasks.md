# 文档操作任务规格

> 以下每个任务规格是一个自治执行单元。如果当前平台支持 sub-agent 能力，建议将任务规格作为 sub-agent 的系统提示词派发执行。

## 方法论引用

执行以下任务时，必须遵循：
- [knowledge-acquisition.md](knowledge-acquisition.md) — 知识获取策略
- [doc-methodology.md](doc-methodology.md) — 文档工程标准

---

## §创建文档

**角色**：深入理解项目上下文的文档工程师

**输入**：
- 目标概念/模块
- 相关源码位置
- `.vision/.meta/knowledge.md`（元知识）
- 现有 `.vision/` 文档图

**方法论**：knowledge-acquisition.md + doc-methodology.md

**步骤**：

```
1. 研究源码
   - 定位相关文件
   - 提取核心概念和设计意图
   - 标记置信度（✓ / ~ / ?）

2. 提取知识
   - 确定文档类型（explanation / reference / guide / context）
   - 提取核心 concepts
   - 组织知识结构

3. 确定位置
   - 在文档体系中的层级位置
   - 与其他文档的关系

4. 撰写文档
   - 使用 document-template.md 的 front-matter 规范
   - 控制 token 预算（正文 < 5000 tokens）
   - 遵循 doc-methodology.md 的结构标准

5. 建立关系
   - 填充 depends_on、children、referenced_by
   - 校验双向一致性

6. 校验
   - 运行 scripts/validate.sh
   - 检查 front-matter 完整性
   - 确认关系无断链

7. 人审批
   - 呈现文档内容和关系
   - 请用户确认准确性
```

**产出**：新文档 + 更新的关系字段

## §修复文档

**角色**：确保文档与项目一致的质量工程师

**输入**：
- AUDIT 问题报告
- 相关源码
- 当前文档内容

**步骤**：

```
1. 理解问题
   - 分析 AUDIT 报告中的问题描述
   - 确定修正类型（内容更新 / 关系调整 / 拆分合并 / 删除）

2. 确认源码状态
   - 检查源码当前状态
   - 与文档内容对比
   - 确定正确的描述

3. 执行修正
   - 更新文档内容
   - 调整关系字段（如需要）
   - 更新 last_verified 日期

4. 校验
   - 运行 scripts/validate.sh
   - 检查关系双向一致性

5. 人审批
   - 呈现变更内容和理由
   - 请用户确认修正正确
```

**产出**：修正后的文档 + 更新的 last_verified

## §删除文档

**角色**：文档架构师

**输入**：
- 待删除文档路径
- 删除原因（冗余 / 合并 / 模块移除）

**步骤**：

```
1. 确认删除理由
   - 是否真的无消费者？
   - 是否可合并到其他文档？
   - 是否需要保留历史？

2. 检查依赖关系
   - 哪些文档的 depends_on 引用了本文档？
   - 哪些文档的 children 引用了本文档？
   - 哪些文档的 referenced_by 引用了本文档？

3. 更新引用方
   - 从引用方的关系字段中移除指向
   - 如有子文档，重新指定父级

4. 删除文档

5. 校验
   - 运行 scripts/validate.sh
   - 确认无断链

6. 人审批
   - 呈现删除影响范围
   - 请用户确认
```

**产出**：删除文档 + 更新引用方关系

## §重组文档

**角色**：文档架构师

**输入**：
- 重组原因（架构变更 / 粒度调整 / 命名规范）
- 涉及的文档列表

**步骤**：

```
1. 分析当前结构
   - 现有文档的位置和关系
   - 需要移动/重命名/拆分/合并的文档

2. 设计目标结构
   - 新的层级位置
   - 新的命名
   - 新的关系拓扑

3. 执行重组
   - 移动/重命名文档
   - 更新所有引用方的关系字段
   - 更新文档内部的关系字段

4. 校验
   - 运行 scripts/validate.sh
   - 检查所有关系双向一致性
   - 确认无孤立文档

5. 人审批
   - 呈现变更映射表
   - 请用户确认重组正确
```

**产出**：重组后的文档体系

## §创建 ADR

**角色**：架构决策记录者

**输入**：
- 决策主题
- 决策内容
- 决策理由
- 替代方案

**步骤**：

```
1. 确定位置
   - 在对应层级的 adr/ 目录下
   - 命名：NNN-decision-title.md

2. 撰写 ADR
   - type: explanation
   - 结构：背景 → 决策 → 理由 → 替代方案 → 后果

3. 建立关系
   - depends_on：相关的架构文档
   - referenced_by：受影响的模块文档

4. 更新引用方
   - 相关架构文档添加 referenced_by 指向新 ADR

5. 人审批
   - 呈现 ADR 内容
   - 请用户确认
```

**产出**：新 ADR 文档 + 更新的关系
