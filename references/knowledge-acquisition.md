# 知识获取与管理方法论

本文件定义 vision-maker 如何从不同来源获取、验证和管理项目知识。

## 研究策略

### 源码提取

从项目源码中定位和提取知识：

1. **代码结构分析**：目录层级、模块划分、命名约定
2. **接口定义**：API 端点、函数签名、数据结构
3. **注释与文档字符串**：内联的设计意图和使用说明
4. **配置文件**：技术栈、依赖关系、环境约束
5. **测试用例**：预期行为、边界条件、使用场景

### 文档消费

从已有文档中提取知识：

1. **README / CONTRIBUTING**：项目定位、使用方式、贡献规范
2. **ADR（架构决策记录）**：设计决策及其理由
3. **Wiki / 外部文档**：补充的背景知识和教程
4. **CHANGELOG**：演进历史和重要变更

## 追溯策略

### Git History 追溯

从版本控制历史中恢复设计决策：

```
1. 定位相关文件的变更历史（git log --follow <file>）
2. 分析 commit message 中的设计意图
3. 查找关联的 PR / Issue（通过 commit message 中的引用）
4. 追溯分支合并历史，理解特性演进脉络
```

### PR / Issue 追溯

从代码审查和问题跟踪中恢复决策上下文：

```
1. 通过 commit message 中的 PR 编号获取 PR 详情
2. 分析 PR 描述、审查评论、讨论结论
3. 关联的 Issue 提供问题背景和需求来源
4. 标签和里程碑提供优先级和版本信息
```

## 置信度框架

对获取的知识标记置信度，帮助使用者判断可靠性：

| 标记 | 含义 | 适用场景 |
|------|------|---------|
| ✓ 已验证 | 通过代码/文档/对话确认 | 源码直接可验证、用户明确确认 |
| ~ 推断 | 基于上下文合理推断 | 命名约定推断、模式推断 |
| ? 未知 | 无法确认，需要进一步核实 | 文档过时、代码与文档矛盾 |

### 置信度更新规则

- 新获取的知识默认标记为 `~ 推断`
- 通过多源验证（代码 + 文档 + 用户确认）可提升为 `✓ 已验证`
- 发现矛盾或过时信息时降级为 `? 未知`
- 建议用户将 `~ 推断` 和 `? 未知` 的知识正式确认后写入 `.vision/`

## 对话获取策略

### 何时问用户

- 源码中找不到明确的设计意图
- 发现多个可能的解释，需要用户确认
- 涉及业务逻辑和领域术语的理解
- 需要了解团队规范和约定
- 新功能/模块的设计决策

### 如何提问

遵循用户适配原则：

1. **每次只问一个问题**——降低认知负担
2. **优先使用多选题**——比开放题更容易回答
3. **给出明确推荐和理由**——帮助用户快速决策
4. **将用户散碎的回答组织为结构化内容**——确认是否准确
5. **主动识别用户未提及但可能重要的方面**——暴露盲区

### 提问模式

```
模式 1：确认理解
"我从代码中看到 [观察]，这是否意味着 [推断]？"

模式 2：二选一
"关于 [主题]，有两种常见做法：
 A. [方案A + 理由]
 B. [方案B + 理由]
 你倾向哪种？我推荐 A，因为 [推荐理由]。"

模式 3：补充盲区
"你提到了 [已提及内容]，关于 [可能遗漏的内容]，有什么补充吗？"
```

## 用户画像管理

### 画像模型

通过对话持续刻画用户的知识面和认知特征：

**特征 1：知识面有限，存在认知盲区**
- 默认将用户视为新手
- 用户提出的问题 → 暴露未知领域
- 用户输入的知识 → 确认已知领域
- 根据知识面决定详略

**特征 2：思维散点化**
- 主动将零散想法组织为结构化认知面
- 连接点与点之间的关系

**特征 3：方法论了解有限**
- 主动上升到方法论层面解释和引导
- 解释"为什么这样做"，而不仅是"做什么"

**特征 4：记忆有限，上下文易丢失**
- 输出时始终补充完整上下文
- 跨会话时主动帮助恢复上下文

**特征 5：决策疲劳**
- 精简选项（2-3 个）+ 明确推荐 + 理由

**特征 6：锚定偏差**
- 主动暴露盲区和替代视角
- 用"是否考虑过..."引导

**特征 7：领域专家 ≠ 文档专家**
- 通过提问引导用户表达隐性知识
- 将口述内容转化为结构化文档

### 用户画像存储

存储在 `.vision/.meta/user.local.md`，不纳入版本控制。

```yaml
---
last_updated: YYYY-MM-DD
---
```

**知识面画像**
- 已确认的知识领域（附证据）
- 已识别的知识盲区（附证据）
- 详略调整策略

**沟通偏好**
- 偏好的沟通风格
- 明确提出的工作习惯

**交互记录摘要**
- 关键讨论结论的简要索引
- 用于跨会话恢复上下文

### 更新时机

- INIT 阶段：初始化画像
- 每次交互：根据新证据增量更新
- 不需要用户主动维护——Skill 自动管理

## 知识回流规则

在任务执行过程中获得的新知识，应流回 `.vision/` 文档体系：

1. **即时补充的知识**（BRIEF 过程中发现的盲区）
   - 标记置信度
   - 建议用户正式写入 `.vision/`
   - 如用户确认，引用 doc-tasks.md 执行创建/更新

2. **审计发现的偏差**（AUDIT 过程中发现的不一致）
   - 记录在审计报告中
   - 引用 doc-tasks.md 执行修复
   - 更新 last_verified

3. **对话中获取的新知识**
   - 评估是否应写入 .vision/
   - 如是，建议对应的目标文档和位置
